		- Similarly, we should be able to press left to accept the offer and right to decline (or similar), but still allow the player the ability to click the buttons. This will give the added benefit of dead-simple gamepad support
		- When a dialogue box appears, there should be some brief pauses to allow the players eyes a chance to catch up. Here is the ideal flow:
			- Dialogue box enters, name section is already pre-filled
				- Please add juice to this by having the box overshoot its destination slightly before going to its intended position
				- Please do the same in reverse for when the box leaves: have it move down slightly before sliding back up
				- You can do this by tweening from current position to target position, I really recommend getting familiar with tweens if you are not already, they're incredibly powerful
				- Please create an export var of tween transition types for both in and out animations so we can see which works best
				- This will change the dialogue box from using an animation player to using tweens, which is a big design revision but I think it will pay off in the end
				- Please also create export variables to let us choose the following:
					- How far the box overshoots
					- How long the brief pauses are
					- How long to pause between each letter
			- Very brief pause, maybe 0.15-0.2s
			- Text begins to fill the dialogue box
			- Do not show the continue button until the dialogue is finished
				- Allow the player to press E (or similar) to force the dialogue to complete if they want to mash through
		- Dialogue Box flow
			- The dialogue box should not go away between the greeting text and haggling
		- Debug UI

				- Day phase (Morning, Day, Night)

Code:
	7/6
		- The villager manager is handling a lot, it handles the haggling, the generation of offers, the acceptance/rejections, even whether or not the shop is open. I fear that it will become kind of unwieldy if left unchecked and there is a lot of dependency on this one script if things go this way which I think we could (or should) avoid.
		- My personal recommendation here is to split it into:
			VillagerManager:
				- Handles the spawning of villagers
				- Who can spawn
				- Who is alive or dead so that other villagers can reference them
			HaggleManager:
				- Generates offers based on info given and info possibly taken from a world-state global
			ShopManager:
				- Set the store open/closed status here
				- Let this handle acceptance/rejection, that kind of thing
			That kind of thing. But in general my mantra with scripts has been to just have each script responsible for one thing. Easier to fix. Easier for me to review. Which I know is a lot to ask during a jam haha, I promise I'm not asking you to write production-level code, just my personal architectural choices.
		- Villager data should probably be a custom resource, it'll be easy to make changes and slot things in on a per-villager basis. It will hold their name, their sprite frames, their wealth level, their dialogue options, that kind of thing.
		- That JSON is going to get wild considering I have about 100 or more lines to write, all with differing conditions. As the writer who will need to slot things into where they need to go, I would ask that you please change to a custom resource based dialogue system where we can define who can say what and under which conditions they can say it and then slot it into villagers in their individual custom resources (see above) that hold their info. Otherwise, if we come up with a new condition it is gonna be a mammoth task trying to figure out how to implement this.
		- In general, I am a huge proponent of using custom resources. Godot is basically built around them and the node system it uses. I didn't ask to see your portfolio before you came aboard so I'm not sure what your confidence level is around them, but if you haven't made any big projects in Godot I think it's a great way to build scalable architecture. I use them for basically everything data-related. I'd be more than happy to walk you through it!
